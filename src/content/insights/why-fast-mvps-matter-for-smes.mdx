## A first release should answer a real question

An MVP is useful when it gives the business evidence. It should not be a thin version of a huge imagined platform. It should be the smallest credible system that proves whether a better way of working is worth pursuing.

For an SME, that distinction matters. Time, budget, and attention are limited. The first release needs to earn the right to become a larger build.

<Callout title="A sensible MVP question">
  Can we put one useful workflow in front of real people quickly enough to learn what should happen next?
</Callout>

## Narrow scope creates useful speed

A focused MVP may include:

- One clear user journey.
- The smallest reliable data model.
- A controlled set of integrations.
- A simple operational view for the team.
- Enough measurement to understand what happened.

It should not quietly collect every possible feature request before the first release has been tested.

<BuildMetric label="Planning principle" value="One useful release" note="Build the smallest credible route to evidence, then improve it." />

<StudioNote>
  This approach works for customer-facing products and internal systems. A quoting workflow, client portal, or enquiry tool can all start with one reliable operational improvement.
</StudioNote>

## Speed is not the same as haste

Fast delivery still needs clear interfaces, sensible validation, and ownership after launch. The aim is to reduce avoidable delay while keeping the software straightforward to extend.

<QuoteBlock>
  The first build should create evidence, not a backlog disguised as certainty.
</QuoteBlock>
